## Context

LinaPro 当前定位是面向可持续交付的 AI 原生全栈框架，已经具备核心宿主、管理工作台、源码插件、动态插件、租户、权限、i18n、缓存和集群协调等基础能力。现有插件 HTTP 能力主要挂载到后台管理 API 面，默认复用后台用户 JWT、角色菜单权限和数据权限语义。

本变更聚焦轻量 C 端承载能力：一个 C 端业务服务由一个业务插件完整提供。商城、内容站、客户门户等业务可以作为垂直插件交付，插件内部可以自行拆分商品、订单、会员等模块，但对宿主保持一个独立业务单元。宿主不在第一期引入 Consumer App Registry、多插件组合或服务编排模型。

本轮边界重新收束后，该能力统一称为“插件消费者服务面”：插件公开消费者服务时使用的宿主接入与治理层，而不是 LinaPro 内建的 C 端产品运行时。宿主只负责让插件可以安全、稳定、可观测地公开 C 端 API 和可选前端资源；账号体系、登录实现、商品、内容、订单、会员、业务后台和业务权限仍属于插件自己的产品边界。

## Goals / Non-Goals

**Goals:**

- 建立独立于后台 `/api/v1` 的 C 端 API 服务面 `/api/c/v1`。
- 允许源码插件按 `/api/c/v1/<plugin-id>/...` 注册 C 端 API。
- 提供 C 端请求上下文和租户边界中间件契约。
- 让 C 端接口纳入插件启用状态、租户、i18n、缓存、错误响应和 API 文档治理。
- 允许业务插件可选托管自己的 C 端前端资产。
- 明确后台管理员用户和 C 端消费者主体不得被强制混用。
- 固化宿主插件消费者服务面与业务插件的责任边界，避免宿主演进为商城、CMS、官网或统一 C 端用户中心等具体产品。

**Non-Goals:**

- 不实现具体商城、会员、商品、订单、支付或内容业务。
- 不引入多插件组成一个 App 的运行时编排。
- 不要求多个业务插件共享一套 C 端用户池。
- 不把现有 `sys_user` 改造成唯一 C 端用户表。
- 不要求第一期完整支持动态插件 C 端路由执行，只预留 route surface 扩展方向。
- 不改变现有后台管理 API、后台登录和 RBAC 权限行为。
- 不在当前阶段实现插件自定义 Consumer Auth Provider 注入、Consumer Tenant Provider、统一限流策略或公网安全防护策略；这些能力需要在插件消费者服务面边界稳定后单独评估。
- 不把商品管理、内容管理、会员管理、订单管理、门户登录、CMS 登录或官网页面编排纳入宿主通用能力。

## Decisions

### Decision 1: 一个 C 端业务服务由一个业务插件承载

业务插件是 C 端服务的隔离边界。插件可以在内部维护多个业务子模块，但宿主只按插件粒度管理安装、启用、禁用、升级、租户开通、API 前缀和前端资产。

替代方案是引入 Consumer App Registry，让多个插件组成一个 C 端 App。该方案更灵活，但会带来插件组合、跨插件权限、跨插件数据一致性、前端装配和后台 App 作用域治理，超出当前阶段需要。

### Decision 1A: 宿主只提供接入与治理，不提供 C 端产品模型

宿主 C 端能力只覆盖服务面、请求上下文、租户边界、前端资源托管、资源索引、治理快照、API 文档和生命周期失效。宿主不得定义统一消费者账号表、统一商品/内容/订单模型、统一 C 端后台菜单、统一 C 端业务权限模型、Consumer Auth Provider 或默认 C 端 JWT。

业务插件必须自主管理消费者账号、登录、会话、token、消费者主体和业务授权。宿主不参与 C 端 token 签发、解析、撤销或主体注入，避免把插件账号体系和宿主启动编排耦合在一起。

### Decision 2: C 端 API 使用固定服务面 `/api/c/v1/<plugin-id>/...`

后台管理 API 继续使用 `/api/v1`，C 端 API 使用 `/api/c/v1`。插件 ID 必须出现在 C 端 API 前缀中，确保路由治理、文档分组、缓存作用域、审计归属和插件启用检查都有明确归属。

替代方案是允许插件自定义任意 C 端路径。该方案会提升自由度，但会削弱治理一致性，容易出现路径冲突、文档混乱和插件不可用时降级不一致。

### Decision 3: C 端身份由插件自理，宿主只传递请求元数据

宿主提供请求上下文能力，例如 `tenantId`、`pluginId`、`anonymousId`、`deviceId`、`channel`、`locale`。具体消费者账号、会员资料、等级、积分、标签、登录态、session、token 和业务身份主体由业务插件自己维护。

后台 `sys_user` 仍表示管理员和后台操作者。C 端消费者可以由插件自有表、外部 IdP 或插件自有账号模块产生，但不得映射为宿主统一 Consumer Principal，因为宿主不再定义该主体模型。

### Decision 4: 宿主不提供 C 端访问级别认证中间件

C 端路由是否匿名、可选登录或强制登录，属于业务插件内部认证语义。宿主不发布 `ConsumerAuthOptional`、`ConsumerAuthRequired` 或等价 Consumer Auth Provider 接缝，也不提供默认 host-signed consumer JWT。插件可以在自己的路由组中挂载插件自有认证中间件，或在 handler/service 内部校验登录态。

后台 RBAC 的 `permission` 语义不直接套用到 C 端。C 端资源授权由插件业务策略和资源归属校验承担，宿主只保证请求已经带有插件 ID、租户、语言、匿名标识、设备和渠道等基础元数据。

### Decision 5: C 端租户优先在认证前解析

C 端公开接口也可能需要租户上下文，例如公开商品列表、公开内容详情等。C 端租户解析不能完全依赖登录 token，应支持从域名、路径、Header、查询参数或插件自定义解析器中建立 `tenantId`。

当租户能力未启用时，C 端请求退化到平台默认租户边界；当租户能力启用但无法解析租户时，受租户约束的 C 端接口必须失败关闭。

### Decision 6: 插件前台页面可选，宿主只提供托管能力

业务插件可以只提供 C 端 API，让使用者用外部 H5、Next.js、UniApp 或小程序调用。插件也可以提供 `frontend/consumer/` 资产，由宿主托管到稳定路径。

插件后台管理页面继续挂载现有管理工作台。宿主不负责把多个插件的前台页面组装成一个 App，也不要求 C 端页面必须运行在宿主内。

### Decision 7: API 文档必须区分 Admin API 与 Consumer API

OpenAPI 聚合需要按服务面分组，至少能区分后台管理接口与 C 端接口。C 端接口不得只混入现有后台 API 文档分组中，避免用户误用后台 token、后台权限和后台路径。

## Risks / Trade-offs

- C 端身份抽象过薄，短期内不同业务插件可能各自实现账号表和登录方式 → 第一阶段接受业务插件自理，宿主只保证服务面、请求上下文和租户边界稳定，后续如需官方 consumer-account 能力应以独立插件而非宿主内建能力评估。
- 固定 `/api/c/v1/<plugin-id>/...` 前缀降低路径自由度 → 换取插件隔离、文档清晰和治理一致性。
- 一个业务服务一个插件可能导致大型插件内部复杂 → 通过插件内部目录分层和服务边界治理解决，避免过早拆成多个插件造成跨插件编排复杂。
- C 端租户解析早于认证会增加中间件顺序复杂度 → 通过独立 ConsumerCtx/ConsumerTenant 中间件和测试覆盖固定顺序。
- 第一阶段不支持多插件组装 App，某些复用场景需要重复实现或手动集成 → 保持当前阶段简单，等出现明确复用需求后再评估 Consumer App Registry。
- 如果实现 Consumer Auth Provider 注入，容易把插件账号体系、宿主启动编排和插件依赖注入耦合在一起 → 当前阶段从宿主移除该方向，插件自有认证即可满足一个业务服务一个插件的边界模型。

## Migration Plan

1. 保留现有 `/api/v1` 后台管理 API 行为不变。
2. 新增 `/api/c/v1` 路由面和插件注册能力，不迁移现有后台接口。
3. 为示例插件或测试插件增加最小 C 端 API 验证链路。
4. API 文档新增 C 端分组后，现有后台 API 文档继续可用。
5. 插件可逐步选择是否新增 `frontend/consumer/` 资产；没有 C 端资产的插件不受影响。

回滚策略：关闭或移除 `/api/c/v1` 路由注册入口即可，不影响现有 `/api/v1`、插件后台页面和动态插件后台路由。

## Open Questions

- C 端租户解析第一期是否只支持 Header/Query，还是同时支持域名映射？
- 是否需要在第一期提供一个官方 demo consumer plugin，还是只用测试插件覆盖契约？
- C 端审计应先复用现有操作日志事件，还是定义独立 consumer event 接缝？
