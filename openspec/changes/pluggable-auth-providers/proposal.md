## Why

当前系统登录方式以用户名和密码为主，后续接入 QQ、微信、GitHub、企业微信、钉钉、飞书、Google、Microsoft、企业 SSO 等登录方式时，如果逐一固化到宿主服务，会持续扩大认证主链路和配置治理的复杂度。

项目已经具备插件运行、Hook/Slot、插件前后端集成等基础能力，本次变更需要把“外部登录提供方”定义为宿主受控、插件可扩展的认证能力，让宿主继续掌握本地用户、租户、权限、Token、会话和审计边界，而把不同身份源协议适配交给插件实现。

## What Changes

- 新增宿主级可插拔登录提供方治理能力，维护已启用登录提供方的元数据、排序、状态、插件来源和非敏感配置引用。
- 新增外部身份绑定模型，将外部登录身份映射到现有本地用户，首期不自动创建用户、不自动分配租户、不自动分配角色。
- 新增统一外部登录流程：提供方列表、授权跳转、回调处理、本地身份绑定校验、JWT/Refresh Token 签发、在线会话创建和认证生命周期事件发布。
- 新增插件认证提供方契约，由插件返回标准化 `ExternalIdentity`，宿主负责本地账号映射、用户状态校验、租户边界校验、Token 签发和会话持久化。
- 首期交付通用 OIDC 登录插件和企业微信登录插件；Google、Microsoft、Azure AD/Entra ID、Keycloak、Okta 等通过 OIDC 提供方实例配置接入，钉钉、飞书、QQ、个人微信、SAML 等作为后续插件扩展。
- 新增个人中心外部身份绑定/解绑入口，首期支持用户自助绑定已启用的外部登录提供方。
- 调整登录页展示逻辑，在用户名密码登录之外动态展示宿主启用且当前前端可渲染的外部登录入口。
- 敏感密钥首期通过 `config.yaml` 或环境变量维护，数据库仅存储非敏感元数据、公开配置或配置引用。
- OAuth/OIDC 的 `state`、`nonce`、PKCE 等临时状态必须使用宿主统一缓存/协调能力存储，保证集群部署下任一实例均可处理回调。
- 外部登录失败、未绑定、提供方禁用、插件不可用、状态校验失败等场景必须返回稳定业务错误，并参与登录日志和审计链路。

## Capabilities

### New Capabilities

- `pluggable-auth-provider`: 定义宿主可治理、插件可注册的外部登录提供方、外部身份绑定、授权回调和本地登录转换流程。

### Modified Capabilities

- `user-auth`: 用户认证能力从仅用户名密码登录扩展为支持宿主受控的外部身份登录，但 Token、会话、用户状态和权限仍由宿主统一处理。
- `login-page-presentation`: 登录页从“当前阶段仅暴露用户名/密码入口”调整为可动态展示已启用的外部登录入口。
- `plugin-hook-slot-extension`: 后端插件扩展点增加认证提供方注册契约，并明确认证提供方与既有认证 Hook 的边界。
- `plugin-ui-integration`: 插件前端集成需要支持外部登录入口和个人中心绑定入口的宿主页面集成。

## Impact

- 后端认证 API、认证服务、JWT/Refresh Token 签发、在线会话写入、登录生命周期事件、登录日志事件载荷。
- 插件宿主服务、源码插件注册契约、插件启停治理、插件配置读取和插件运行时错误隔离。
- 数据库新增宿主迁移 `013-pluggable-auth-providers.sql`，包含登录提供方元数据表和外部身份绑定表，并生成对应 DAO/DO/Entity。
- 新增源码插件目录：`apps/lina-plugins/auth-oidc/` 与 `apps/lina-plugins/auth-wecom/`，遵循源码插件目录结构规范。
- 前端登录页、个人中心安全设置页、认证 API 客户端、插件 Slot/页面集成和 i18n 资源。
- 安全影响包括 OAuth/OIDC CSRF 防护、nonce 校验、PKCE、回调地址白名单、密钥不落库、外部身份唯一绑定和未绑定账号拒绝登录。
- 缓存影响包括授权临时状态、提供方启用状态缓存、插件状态变化后的缓存失效和集群部署下跨实例一致性。
- 测试影响包括后端单元测试、认证流程集成测试、插件注册契约测试、前端登录入口渲染测试和绑定/解绑 E2E 测试。
